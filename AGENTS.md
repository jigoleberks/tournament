# AGENTS.md

Instructions for AI coding agents working in this repository.

## Commands — everything runs through Docker

All Rails commands go through `docker compose run --rm web`. Never run `rails` or `bundle` directly on the host.

```bash
# Start the app
docker compose up

# Run a one-off Rails command
docker compose run --rm web bin/rails <command>

# Run the full test suite (both unit and system)
docker compose run --rm web bin/rails test
docker compose run --rm web bin/rails test:system

# Run a single test file or specific line
docker compose run --rm web bin/rails test test/models/user_test.rb
docker compose run --rm web bin/rails test test/models/user_test.rb:42
```

**After editing `.env`, run `docker compose up -d`, NOT `docker compose restart`.** `restart` reuses the existing container's environment and won't pick up new values.

## Test conventions

- Minitest with parallel execution, FactoryBot factories in `test/factories.rb`, `fixtures :all`.
- System tests use Cuprite + Chromium. Camera tests rely on headless Chromium flags configured in `test/application_system_test_case.rb`.
- `test/test_helper.rb` provides custom helpers: `with_broadcast_spy`, `count_queries` (N+1 detection), `with_class_method_stub`.
- CI runs `bin/rails db:test:prepare` then `bin/rails test` then `bin/rails test:system`.
- **Re-run the full test suite before any `git push`.** Run `docker compose exec web bin/rails test` after the last code change. Don't push and rely on CI.

## Lint / style

- RuboCop with `rubocop-rails-omakase` (`.rubocop.yml`). Run with `docker compose exec web bin/rubocop`.
- Tailwind utility classes inline in ERB views — no separate CSS files.
- Enums stored as integers.
- Service objects use the `Module::Class` pattern with a `self.call` class method.

## Architecture

- Multi-club: a single deployment hosts multiple `Club`s. Tournaments, templates, and per-club roles are club-scoped; `User`s and `Species` are global.
- Authentication is custom-built (no Devise). Magic-link sign-in via `SignInToken`. Session stores `user_id`.
- Two authorization concepts on `User`:
  - **Per-club role** — lives on `ClubMembership`, enum `member` (0) or `organizer` (1).
  - **Site admin** — boolean `admin` flag on `User`, scope-free.
- Base controllers enforce access: `Organizers::BaseController` (organizer in current club), `Judges::BaseController` (judge assignment for specific tournament), `Api::BaseController` (signed in, returns 401), `Admin::BaseController` (organizer in current club), `Admin::Clubs::BaseController` (site admin).
- Routes: `/organizers/*`, `/judges/tournaments/:tournament_id/*`, `/admin/*`, `/admin/clubs/*` (site admin only), `/api/*`.
- Soft deletes via `deactivated_at` on User, `active: false` on CatchPlacement.
- Core scoring: `Catches::PlaceInSlots` service places catches into tournament scoring slots, broadcasts leaderboard, triggers push notifications.

## Branching

- Day-to-day development on `jig_dev`. Test VMs deploy from `jig_dev`; prod VM tracks `main`.
- Pull before pushing: `git pull --rebase origin jig_dev`.
- One PR per release cut (`jig_dev → main`). Squash-merge is fine.
- Drain open dependabot PRs before opening a release PR.

## Key gotchas

- `DATABASE_URL` in docker-compose omits the database name — let `config/database.yml` append `-development` / `-test` / `-production` per environment. Including the full path in ENV caused `bin/rails test` to truncate the dev DB.
- Solid Queue runs in-process via `SOLID_QUEUE_IN_PUMA: "1"`.
- Catch photos have `client_uuid` for offline deduplication.
- After pulling new code, stale `tmp/pids/server.pid` can prevent Puma from starting — `rm -f tmp/pids/server.pid` before `docker compose up`.
- `app/assets/builds/tailwind.css` is generated by `bin/rails tailwindcss:build` and committed. Docker entrypoint builds it on startup.
